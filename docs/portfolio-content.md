## 06-12. popn.gg Case Study

### Project Intro

- 기간: 2025.03 - 2025.06
- 형태: Team Project
- 역할: Backend Developer
- 핵심 키워드: DTO Projection, Query Optimization, Statistics API, Cloudflare CDN, Image Delivery

플레이데이터 조회와 이미지 리소스 제공 구조를 개선한 리듬게임 플레이데이터 관리 서비스입니다.

많은 플레이 기록을 빠르게 조회하고 분석할 수 있도록 만들었습니다.

DTO Projection, DB 집계, 페이지네이션을 적용해 대량 데이터 조회 비용을 줄이고, S3와 Cloudflare CDN으로 반복되는 이미지 요청을 효율적으로 처리했습니다.

### Service Images

- `assets/projects/popn/home.png`: 홈 / 신규 악곡 화면
- `assets/projects/popn/profile.png`: 유저 프로필 화면
- `assets/projects/popn/analysis.png`: 곡 상세 분석 화면

### Problem & Goal

- Problem: 대량 플레이데이터 조회와 이미지 리소스 제공 과정에서 응답 비용이 커지는 문제가 있었습니다.
- Solution: DTO Projection, DB 집계, 페이지네이션을 적용해 화면에 필요한 데이터만 조회하도록 개선했습니다.
- Goal: 곡 목록, 유저 프로필, 곡 상세 분석 화면에서 대량 데이터를 빠르게 제공하고 이미지 로딩 체감을 개선합니다.

### My Role

- 백엔드 API 설계 및 데이터 조회 최적화
- 통계 응답 구조와 DB 집계 기반 조회 흐름 설계
- S3 원본 저장소와 Cloudflare CDN 기반 이미지 제공 흐름 구성
- 레거시 Entity 중심 조회를 화면 전용 Projection 구조로 재설계

### Tech Stack

- Spring Boot: REST API와 도메인 중심 백엔드 구조를 안정적으로 구성하기 위해 사용
- JPA / JPQL: DTO Projection으로 필요한 필드만 조회하고 Entity 로딩 비용을 줄이기 위해 사용
- MySQL: 랭크, 메달, 레벨별 통계를 DB 집계 중심으로 처리하기 위해 사용
- Redis: 검색용 Read Model과 대량 갱신 Lock을 구성해 응답 크기와 중복 작업을 줄이기 위해 사용
- S3 / Cloudflare CDN: 곡 자켓 이미지 원본 저장과 edge cache 기반 이미지 제공 구조를 만들기 위해 사용

### Implementation

- 대량 조회 성능 개선: Entity 전체 조회를 줄이고 DTO Projection과 페이지네이션으로 필요한 데이터만 조회했습니다.
- 통계 집계 구조 개선: 랭크, 메달, 레벨별 통계를 애플리케이션 순회가 아닌 DB 집계와 Projection 중심으로 처리했습니다.
- 이미지 제공 구조 개선: S3 원본 저장소 앞단에 Cloudflare CDN을 구성해 반복 이미지 요청을 edge cache에서 처리했습니다.
- 라이브 검색 최적화: 전체 데이터를 내려받아 검색하던 구조를 백엔드 검색 API와 Redis Read Model 기반으로 전환했습니다.
- 대량 갱신 안정화: playdata 갱신을 요청 흐름에서 분리하고 Redis Lock과 Worker 기반 처리 구조로 정리했습니다.

### Troubleshooting

- Entity 전체 조회와 Lazy Loading으로 인한 N+1 가능성
  - 원인: 화면 응답에 필요한 값보다 더 큰 Entity 그래프를 조회하는 레거시 구조
  - 해결: 화면 전용 DTO Projection 조회로 변경하고, 페이지네이션을 적용해 조회 범위를 제한
  - 결과: 불필요한 객체 생성과 반복 조회 가능성을 줄이고 대량 playdata 조회 흐름을 단순화
- 곡 자켓 이미지 반복 요청으로 인한 로딩 지연과 원본 요청 부하
  - 원인: 곡 목록, 검색, 랭킹 화면에서 동일 이미지가 반복 조회되는 서비스 특성
  - 해결: S3에 저장된 곡 자켓 이미지를 Cloudflare CDN으로 제공해 edge cache에서 처리
  - 결과: 이미지 제공 책임을 분리하고 반복 조회 화면의 이미지 로딩 체감을 개선

### Result & Learning

- Result: 대량 데이터 조회 흐름 단순화, 통계 API 조회 비용 감소, 곡 자켓 이미지 제공 책임 분리
- Learning: 화면 요구사항에 맞는 Projection 설계, DB 집계와 애플리케이션 처리 책임 분리, CDN cache를 고려한 이미지 제공 구조
- Next: 실제 트래픽 기반 성능 측정, Redis 검색 Read Model 고도화, 대량 갱신 Worker 모니터링 강화

## 13-19. Cloud# Case Study

### Project Intro

- 기간: 2025.07 - 2025.09
- 형태: Team Project
- 역할: Backend Developer
- 핵심 키워드: Collaboration, File Upload, Permission, Self-hosting

Cloud#은 팀 단위 협업 공간을 중심으로 파일을 관리하는 셀프호스팅 서비스입니다.

### Problem & Goal

- Problem: 개인 드라이브와 공유 폴더 중심 구조는 팀 협업, 권한 변경, 용량 정책이 얽히며 관리가 어려워질 수 있습니다.
- Why: 사용자는 파일을 업로드하는 것뿐 아니라 누가, 어디서, 어떤 권한으로 접근하는지도 함께 관리해야 합니다.
- Goal: Space를 최상위 협업 단위로 두고 파일과 멤버, 권한, 저장소 정책을 일관된 흐름으로 연결합니다.

### My Role

- Space 중심 도메인 모델 설계
- 파일 업로드 흐름과 저장소 책임 분리
- 멤버와 권한 정책의 기본 구조 설계
- 셀프호스팅 환경을 고려한 백엔드 구성

### Tech Stack

- NestJS: 모듈 단위로 기능을 분리하고 TypeScript 기반 서버 구조를 명확하게 관리하기 위해 사용
- tus: 대용량 파일 업로드 중단과 재개를 고려한 업로드 흐름을 만들기 위해 사용
- PostgreSQL: Space, Member, File, Permission 관계를 안정적으로 표현하기 위해 사용
- Docker / Nginx: 셀프호스팅 배포와 프록시 구성을 쉽게 재현하기 위해 사용

### Implementation

- Space 중심 도메인: 파일과 멤버, 권한을 Space 아래로 묶어 협업 단위를 명확히 했습니다.
- 재개 가능한 업로드: 대용량 파일 업로드 중단 상황을 고려해 업로드 세션과 파일 저장 흐름을 분리했습니다.
- 논리/물리 저장소 분리: 사용자가 보는 파일 구조와 실제 저장 위치를 분리해 저장소 변경 가능성을 열어두었습니다.
- 권한 정책 기반 접근: Space 멤버 역할을 기준으로 파일 접근과 관리 권한을 제어하는 구조를 설계했습니다.

### Troubleshooting

- 파일 업로드 중 네트워크가 끊기면 처음부터 다시 올려야 하는 문제
  - 원인: 단일 요청 기반 업로드는 중단 지점을 서버가 추적하기 어렵기 때문
  - 해결: 업로드 세션을 분리하고 재개 가능한 업로드 프로토콜을 적용
  - 결과: 대용량 파일 업로드 실패 비용을 줄일 수 있는 구조 확보
- 파일 권한이 폴더, 멤버, Space 정책과 섞이는 문제
  - 원인: 권한 기준이 여러 계층에 흩어지면 접근 판단 흐름이 복잡해짐
  - 해결: Space를 기준으로 멤버 역할과 파일 접근 흐름을 정리
  - 결과: 협업 단위별 권한 판단 경로가 단순해짐

### Result & Learning

- Result: Space 중심 협업 모델 설계, 재개 가능한 파일 업로드 구조 마련, 저장소 확장 가능성 확보
- Learning: 파일 서비스의 핵심은 저장소보다 권한 흐름이라는 점, 대용량 업로드에서 실패 복구 설계의 중요성, 셀프호스팅 환경에서 배포 재현성이 갖는 가치
- Next: 권한 상속 정책 구체화, 스토리지 어댑터 확장, 사용량 통계와 quota 정책 고도화
